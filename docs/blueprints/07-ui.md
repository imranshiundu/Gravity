# 07 — Gravity UI Blueprint

The UI is where Gravity becomes understandable.

It should not be a random chat page. It should be a command center for Grav, memory, tools, approvals, modules, services, and business workflows.

## One-line definition

**Gravity UI is the shell where users talk to Grav, see active modes, approve risky actions, inspect memory, view logs, and operate modules.**

## Main UI areas

### 1. Grav Chat

The main assistant surface.

Features:

- text chat
- streaming responses
- mode indicators
- tool-call visibility
- memory references
- quick actions

### 2. Mode switcher

Users should know what mode Grav is using.

Modes:

```text
Personal
Business
Coding
System
Memory
Defense
Gateway
Channels
Voice
```

Mode can be automatic or manual.

### 3. Approval panel

Dangerous actions should appear here.

Example approval card:

```text
Action: code.editFile
Risk: medium
File: README.md
Reason: update architecture docs
Approve / Reject
```

### 4. Memory browser

Users can inspect what Grav remembers.

Features:

- search memories
- view source
- edit memory
- delete memory
- mark wrong
- filter by project/customer/mode

### 5. Tool execution log

Shows what Grav did.

Example:

```text
12:01 memory.search completed
12:02 code.scanRepo completed
12:03 code.editFile waiting for approval
12:04 user approved
12:05 code.editFile completed
```

### 6. Module health

Displays service/module status.

```text
Core: online
Ollama: online
Memory: online
Coding: ready
Defense: ready
Gateway: online
Voice: not configured
Channels: disabled
```

### 7. Workspace selector

Gravity should support different contexts.

Examples:

```text
Personal
Gravity
Orb21
Pit Performante
Client A
Business Demo
```

### 8. Business dashboard

Later UI for:

- leads
- bookings
- quote drafts
- customers
- follow-ups
- daily report
- knowledge base

## Design principles

Gravity UI should feel:

- clean
- serious
- technical but not ugly
- fast
- local-first
- readable
- not AI-template generated
- command-center-like

## Important UI rule

The UI must show when Grav is acting.

Users should never wonder:

```text
Did Grav read a file?
Did Grav change something?
Did Grav run a command?
Did Grav use memory?
Did Grav send anything externally?
```

## Animation ideas

Use subtle motion only.

Examples:

- mode badge transitions
- tool-call timeline pulses
- service health status fade
- approval card slide-in
- memory retrieval shimmer
- voice waveform only in Voice Mode

Avoid gimmicks.

## Acceptance tests

UI is working when:

- user can chat with Grav,
- active mode is visible,
- risky actions create approval cards,
- tool calls are visible,
- memory can be inspected,
- module health is displayed,
- UI communicates what Gravity is doing.

## Final blueprint

```text
UI = Gravity command center, not just chat.
```
